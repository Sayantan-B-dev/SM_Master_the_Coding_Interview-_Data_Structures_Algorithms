# Quadratic Time Complexity and Nested Loops: O(n²) Notation

## Abstract

This document introduces the concept of quadratic time complexity, denoted as O(n²), within the framework of Big O asymptotic analysis. The discussion focuses on algorithms involving nested iterations, wherein each element of an input collection is paired with every other element. The classic "log all pairs of an array" problem is presented as a canonical example. The mathematical derivation of O(n²) from nested O(n) operations is explained, and the relationship to Rule 3 (Different Terms for Inputs) is extended to the multiplicative case of O(a * b). The document concludes with a practical heuristic for distinguishing additive and multiplicative complexity patterns based on code indentation.

## 1. Introduction to Quadratic Time Complexity

### 1.1 Definition

**Big O Notation O(n²)** describes an algorithm whose performance—measured by the number of elementary operations—grows proportionally to the **square** of the input size `n`. In formal terms, there exists a positive constant `c` such that for sufficiently large `n`, the number of operations `T(n)` satisfies:

```
T(n) ≤ c * n²
```

Quadratic time complexity is commonly encountered in algorithms that employ **nested loops**, where one loop iterates over the input and, for each iteration, another loop also iterates over the input (or a significant portion thereof).

### 1.2 Intuitive Understanding

If an O(n) algorithm takes 1,000 operations for an input of size 1,000, an O(n²) algorithm performing the same task would require approximately **1,000,000** operations. Doubling the input size from `n` to `2n` results in a **fourfold increase** in the number of operations. This rapid growth rate places O(n²) algorithms in the "Horrible" to "Bad" region of the scalability spectrum.

## 2. The Classic Interview Problem: Log All Pairs of an Array

### 2.1 Problem Statement

Given an array of elements, the task is to generate and log every possible ordered pair `(element_i, element_j)`, including pairs where `i = j`. For an array `['a', 'b', 'c', 'd', 'e']`, the expected output includes pairs such as `(a, a)`, `(a, b)`, `(a, c)`, ..., `(e, d)`, `(e, e)`.

### 2.2 Algorithm Implementation Using Nested Loops

The most straightforward solution involves iterating over the array twice in a nested fashion. The outer loop selects the first element of the pair, and the inner loop selects the second element.

```javascript
/**
 * Logs all possible ordered pairs of elements from the input array.
 * Includes pairs where both elements are the same (i.e., (x, x)).
 * @param {Array} array - The input array.
 */
function logAllPairsOfArray(array) {
    // Outer loop: iterates over each element as the first item of the pair
    for (let i = 0; i < array.length; i++) {
        // Inner loop: iterates over each element as the second item of the pair
        for (let j = 0; j < array.length; j++) {
            // Log the current pair (array[i], array[j])
            console.log(array[i], array[j]);
        }
    }
}

// Example usage
const boxes = ['a', 'b', 'c', 'd', 'e'];
logAllPairsOfArray(boxes);
```

### 2.3 Execution Trace and Operational Count

For an input array of size `n = 5`:

- The outer loop executes **5** times (for `i = 0` to `4`).
- For **each** iteration of the outer loop, the inner loop executes **5** times (for `j = 0` to `4`).

**Total Number of Console Log Operations:** `n * n = n² = 5 * 5 = 25`

Generalizing for an arbitrary input size `n`:

| Input Size (`n`) | Outer Loop Iterations | Inner Loop Iterations per Outer Iteration | Total Operations |
| :--- | :--- | :--- | :--- |
| 2 | 2 | 2 | 4 |
| 3 | 3 | 3 | 9 |
| 4 | 4 | 4 | 16 |
| `n` | `n` | `n` | **n²** |

## 3. Derivation of O(n²) Complexity

### 3.1 The Multiplication Rule for Nested Operations

When loops are **nested** (i.e., one loop resides inside the body of another), the total number of operations is the **product** of the number of iterations of each loop. This contrasts with **sequential** loops, where the operations are **added**.

| Loop Structure | Pattern | Complexity Calculation | Result |
| :--- | :--- | :--- | :--- |
| Sequential | Loop 1; Loop 2; | O(n) + O(n) = O(2n) | **O(n)** |
| Nested | Loop 1 { Loop 2 { } } | O(n) * O(n) | **O(n²)** |

### 3.2 Formal Derivation

The `logAllPairsOfArray` function contains two loops, both of which depend on the same input size `n = array.length`.

- **Outer Loop Complexity:** O(n)
- **Inner Loop Complexity:** O(n)
- **Total Complexity:** O(n) * O(n) = **O(n²)**

Applying the simplification rules:
- No additive constants are present.
- The coefficient is implicitly `1`; the exponent `2` is **not** a constant multiplier—it defines the growth class and is therefore retained.

### 3.3 Graphical Representation

The quadratic growth curve exhibits a parabolic shape, accelerating upward as `n` increases.

```
Operations
    ^
    |                              .
    |                            .
    |                         .
    |                      .
    |                   .
    |                .
    |             .
    |          .
    |       .
    |    .
    | .
    +-----------------------------------> Input Size (n)
```

**Comparison to Linear Growth (O(n)):**
- **Linear (O(n)):** Straight line; slope remains constant.
- **Quadratic (O(n²)):** Curved line; slope increases with `n`.

For large values of `n`, the O(n²) curve rapidly diverges from the O(n) curve, rendering the algorithm impractical for substantial datasets.

## 4. Extension of Rule 3: Different Inputs in Nested Loops

### 4.1 Scenario: Two Independent Arrays

Rule 3 (Different Terms for Inputs) applies equally to nested loops involving distinct input collections. Consider a function that pairs every element from `arrayA` with every element from `arrayB`.

```javascript
/**
 * Logs all pairs formed by taking one element from arrayA and one from arrayB.
 * @param {Array} arrayA - First input array.
 * @param {Array} arrayB - Second input array.
 */
function logCrossPairs(arrayA, arrayB) {
    for (let i = 0; i < arrayA.length; i++) {
        for (let j = 0; j < arrayB.length; j++) {
            console.log(arrayA[i], arrayB[j]);
        }
    }
}
```

### 4.2 Complexity Analysis with Two Variables

Let:
- **a** = `arrayA.length`
- **b** = `arrayB.length`

The outer loop executes `a` times. For each outer iteration, the inner loop executes `b` times.

**Total Operations:** `a * b`

**Final Big O Classification:** **O(a * b)**

### 4.3 Common Mistake: Conflating Input Sizes

It is incorrect to express the complexity of `logCrossPairs` as O(n²) unless `a` and `b` are known to be equal or proportional. The expression `O(a * b)` accurately captures the dependence on two independent input dimensions.

## 5. Heuristic for Identifying Complexity Patterns in Code

A simple, practical rule of thumb can be applied during code inspection to determine whether to add or multiply complexities.

> **Indentation Heuristic:**
> - **Same Indentation Level:** Operations that occur sequentially at the same nesting level are **added**.
> - **Nested Indentation:** Operations that occur within an inner block (deeper indentation) are **multiplied**.

### 5.1 Visual Example

```javascript
function exampleHeuristic(input) {
    // Block A (Indentation Level 1): O(1) or O(n) depending on content
    let x = 5;                        // O(1)

    // Block B (Indentation Level 1): Loop 1
    for (let i = 0; i < input.length; i++) {   // O(n)
        console.log(input[i]);
    }

    // Block C (Indentation Level 1): Loop 2
    for (let i = 0; i < input.length; i++) {   // O(n)
        // Block D (Indentation Level 2): Nested inside Loop 2
        for (let j = 0; j < input.length; j++) { // O(n) * O(n) = O(n²)
            console.log(input[i], input[j]);
        }
    }
}

// Total Complexity: O(1) + O(n) + O(n²) → O(n²) (after applying Rule 4 - Drop Non-Dominants)
```

### 5.2 Application to Multiple Inputs

When multiple inputs are involved, assign distinct variables to each and apply the same heuristic.

```javascript
function processTwoInputs(arr1, arr2) {
    // Sequential (Additive): O(a + b)
    arr1.forEach(item => console.log(item));
    arr2.forEach(item => console.log(item));

    // Nested (Multiplicative): O(a * b)
    arr1.forEach(item1 => {
        arr2.forEach(item2 => {
            console.log(item1, item2);
        });
    });
}
```

## 6. Summary of Quadratic Time Complexity

| Aspect | Description |
| :--- | :--- |
| **Notation** | O(n²) |
| **Common Name** | Quadratic Time |
| **Characteristic Pattern** | Nested loops iterating over the same or similarly sized input. |
| **Growth Rate** | Operations increase proportionally to the square of input size. |
| **Scalability Assessment** | Poor for large inputs; quickly becomes impractical. |
| **Interview Significance** | Frequently presented as a naive solution that must be optimized to O(n log n) or O(n). |

---

## 7. Conclusion

Quadratic time complexity, O(n²), represents a significant milestone in the study of algorithmic efficiency. It is most commonly manifested in algorithms employing nested iterations over the primary input. The "log all pairs" example provides a clear and memorable illustration of how O(n) * O(n) yields O(n²).

The rule of thumb—**add for sequential steps, multiply for nested steps**—simplifies the process of complexity assessment in code reviews and interviews. Furthermore, when independent inputs are involved, the principle extends to multiplicative expressions such as O(a * b). Mastery of O(n²) analysis is essential for recognizing performance bottlenecks and serves as motivation for learning more efficient algorithmic paradigms, such as divide-and-conquer and dynamic programming, which reduce quadratic complexity to linearithmic or linear time.
